# Agent Spec

## Overview

Build an AI Meeting-to-Action Agent that transforms a meeting transcript into structured execution artifacts. The system uses a manager agent to coordinate specialist agents for decisions, action items, owners, deadlines, unresolved questions, follow-up email, and project tracker updates.

The first version is a local prototype. It should be prepared for GitHub linkage after completion, but it should not create or push a repository automatically.

## Example Use Cases

- A product planning transcript is submitted and the agent returns meeting type, decisions, action items with owners and due dates, open questions, a follow-up email, and a tracker CSV.
- A transcript omits owners or due dates and the agent marks owners as `Unassigned` and dates as `Unknown` instead of guessing.
- A developer uses sample transcript loader code for QMSum or AMI-formatted data to test the pipeline locally before adding real ingestion.

## Tools Required

- OpenAI model access through LiteLLM-compatible ADK model configuration.
- Local transcript file loader for plain text.
- QMSum sample loader stub for JSON-style meeting records.
- AMI sample loader stub for XML/transcript-style meeting records.
- CSV exporter for project tracker output.

## Constraints & Safety Rules

- Do not infer owners unless explicitly stated in the transcript. Use `Unassigned` when unclear.
- Do not infer deadlines unless explicitly stated in the transcript. Use `Unknown` when unclear.
- Keep unresolved questions separate from decisions and action items.
- Preserve uncertainty in extracted fields instead of fabricating specificity.
- Do not send emails or update external project trackers in the prototype. Generate email text and tracker CSV only.
- Do not clone, download, or ingest QMSum/AMI data automatically. Provide sample loader hooks only.
- Do not create, initialize, or push a GitHub repository without explicit user approval.

## Success Criteria

- The prototype exposes a manager plus specialist-agent architecture.
- The final output includes meeting summary, meeting type, decisions, action items, open questions, follow-up email, and a downloadable/generated CSV tracker artifact.
- Running a local sample transcript produces the requested report format and a CSV file.
- Unit tests validate deterministic formatting, CSV export, and fallback behavior for unknown owners/deadlines without asserting on free-form LLM content.
